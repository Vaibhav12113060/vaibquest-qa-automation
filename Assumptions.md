# QA Evaluation Assumptions

- **Token Handling:** Implemented under the assumption that the JWT is stored in `localStorage` inside the client browser under the key `token` right after a successful login API response.
- **Token Expiration:** The backend token expiration parameter is assumed to be configured for 1 hour. Frontend routes will check validity before loading dashboard states.
- **Input Trimming:** The system is assumed to automatically trim leading and trailing spaces for input strings (like names and emails) before passing data to validation schemas.
- **Rate Limiting Context:** Server tracks client requests based on the IP address provided in the network request header (handling standard headers like `X-Forwarded-For`).
- **Testing Target & Deviations from Specification:** As the assignment did not provide a target web application for testing, a live, self-developed application (`VaibQuest`) was used to perform the end-to-end automation. This application implements all core features described in the assignment (Registration, Login, JWT-based Authorization). The automated tests are written against the actual behavior of this live application. The following points note where its implementation differs from the assignment's specification, and how the tests were adapted accordingly:
  - **No 'Confirm Password' Field:** The registration form does not include a "Confirm Password" field. Therefore, test cases related to mismatched passwords (e.g., REG-03) are not automated as the feature is not present.
  - **Simplified Password Rule:** The application's validation requires a password of at least 6 characters. It does not enforce the assignment's specified rule of 12 characters with mixed letters and numbers. Automation and boundary tests reflect this simpler 6-character rule.
  - **Auto-Login on Registration:** Upon successful registration, the application automatically logs the user in and redirects to the `/dashboard`. This is a deviation from the specified flow of redirecting to the login page. The automation test `should register a new user successfully` validates this enhanced user experience.
  - **Specific Error Messages:** The tests assert the exact error messages returned by the live application's API (e.g., "User not found", "User already exists"), which in some cases differ from the generic examples provided in the assignment documentation.
